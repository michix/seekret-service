= Seekret Service
:toc: true

Often, users store secrets unencrypted, for example in `.envrc` or `.env` files.
Seekret Service aims to let you store secrets encrypted on your hard drive while keeping them easily and securely accessible.

== The Solution

This service exposes usernames, secrets and SSH keys from a KeePass database file through a webservice interface.
This way, they can be queried using `curl`, for example.
For security reasons, data read from the KeePass database is encrypted in memory and access to the values from the KeePass database has to be authorized by the user with a GUI dialog or other means like Touch-ID on Apple devices.

Since Version 1.3.0, Seekret Service can also act as an `ssh-agent`, thus, making also access to SSH keys transparent and secure.

IMPORTANT: Since version 1.2.6, I use the support of various AI models.
I review the code, but you have the right to know this.
For transparency, I document in the changelog which change was created with which AI model.

== Usage

You may want to start the service in background using the following command:

[source,bash]
----
nohup seekret-service --keepass-path "/path/to/keepassfile.kdbx" > /dev/null 2>&1 &
----

For further help please run:

[source,bash]
----
seekret-service --help
----

After the service is started you may access it using curl:

[source,bash]
.Example for requesting the secret of entry `access-keys/GitLab/GitLab-API-Token-Shell`
----
curl --silent "http://localhost:8123/access-keys/GitLab/GitLab-API-Token-Shell/secret"
----

=== Providing the source of the request

Sometimes you want to understand where a request came from.
For this purpose you can add the HTTP-Header `X-Seekret-Source` when requesting something from Seekret Service.
In consequence, the string provided is displayed in the authorization dialog to the user.

[source,bash]
.Request providing a source string describing the origin
----
GITLAB_TOKEN="$(curl --silent -H "X-Seekret-Source: direnv" http://localhost:8123/access-keys/GitLab/GitLab-API-Token-Shell/username")"
----

=== Usage with direnv

In your `.envrc` file of `direnv` you might want to provide a secret in an environment variable the following way:

[source,bash]
.Example for providing a secret in an environment variable using `direnv` and the file `.envrc`
----
SOURCEINFO="direnv"
[ -d ".git" ] && SOURCEINFO="$SOURCEINFO for $(git config --get remote.origin.url)"
GITLAB_TOKEN="$(curl --silent -H "X-Seekret-Source: $SOURCEINFO" "http://localhost:8123/access-keys/GitLab/GitLab-API-Token-Shell/secret")"
----

The same way you may provide the username of a KeePass entry:

[source,bash]
.Example for providing a username in an environment variable using `direnv` and the file `.envrc`
----
GITLAB_TOKEN="$(curl --silent -H "X-Seekret-Source: $SOURCEINFO" "http://localhost:8123/access-keys/GitLab/GitLab-API-Token-Shell/username")"
----

=== Using as an SSH Agent (Linux/macOS)

Seekret Service can optionally act as an SSH agent, serving SSH private keys stored in your KeePass database.
SSH clients (e.g. `ssh`, `git`) can then authenticate using keys from KeePass without ever writing private keys to disk.

Signing requests are only allowed when the user has recently authorized access through the normal HTTP authorization flow (shared timeout).

==== Storing SSH Keys in KeePass

Store the OpenSSH PEM private key in a custom string field named `ssh-key` on the KeePass entry.
If the key is encrypted with a passphrase, put the passphrase in the password field of the same entry.

==== Starting the SSH Agent

Enable the agent with `--enable-ssh-agent` and specify one or more KeePass entry paths with `--ssh-key`:

[source,bash]
.Starting Seekret Service with SSH agent enabled
----
seekret-service \
  --keepass-path "/path/to/keepassfile.kdbx" \
  --enable-ssh-agent \
  --ssh-key "path/to/ssh-key-entry" \
  --ssh-key "another/ssh-key-entry"
----

The agent listens on a Unix domain socket at `$HOME/.seekret-ssh-agent.sock` by default.
You can override the socket path with `--ssh-agent-sock`:

[source,bash]
----
seekret-service \
  --keepass-path "/path/to/keepassfile.kdbx" \
  --enable-ssh-agent \
  --ssh-key "my-ssh-key" \
  --ssh-agent-sock /tmp/my-custom-agent.sock
----

==== Using the SSH Agent

Point the `SSH_AUTH_SOCK` environment variable at the agent socket:

[source,bash]
----
export SSH_AUTH_SOCK="$HOME/.seekret-ssh-agent.sock"
ssh-add -l          # list loaded keys
ssh user@host       # authenticate using a KeePass-stored key
----

NOTE: Signing requests are denied silently when the authorization timeout has expired.

== Testing

=== Unit-Tests

[source,bash]
----
RUST_LOG=debug cargo test -- --nocapture
----

=== Testing with Test Keystore

[source,bash]
----
RUST_LOG=debug cargo run -- --keepass-path test.kdbx --keepass-keyfile test.key --port 8124 --enable-ssh-agent --ssh-key "my-ssh-key"
----

Test-keyfile `test.kdbx` with password `test` and keyfile `test.key`.
Here you can find the entry `root_entry1` wich username `root-username` and password `root-password`.
You may then request username and password of this test entry using:

[source,bash]
----
curl http://127.0.0.1:8124/root_entry1/username
curl http://127.0.0.1:8124/root_entry1/secret
----

== Release

A new release is performed using `./release.sh <version-number>`.

A release build can be created using the following command:

[source,bash]
.Command for creating a release binary
----
cargo build --release
----

== Development

To automatically update all dependencies and run tests:

[source,bash]
----
./update-dependencies.sh
----

This script will:

1. Update all dependencies to their latest compatible versions
2. Build the project
3. Run all tests
4. Run clippy linter
5. Check code formatting
6. Display a summary of dependency changes

After running the script, review the changes and update `CHANGELOG.md` if needed.

* Install `cross`: `cargo install cross`
* Install targets for Windows:
** `rustup target add x86_64-pc-windows-gnu`
** `rustup target add x86_64-pc-windows-msvc`
